2 Worker Loop — главный цикл

Sleep / Poll Timer

Воркер спит PollIntervalSeconds (30 сек по умолчанию), затем просыпается.

Acquire Lease

Атомарный findAndModify в MongoDB: если lease на шард свободен или истёк — захватываем. Если занят другим воркером — пропускаем итерацию.

Load Checkpoint

Читаем ShardCheckpoint из MongoDB — узнаём last_completed_bucket_end_utc. Если чекпоинта нет (первый запуск) — начинаем с now - InitialLookbackHours.

Compute Pending Windows

Рассчитываем окна от чекпоинта до floor5min(now - SafetyLag).
Ограничиваем MaxWindowsPerTick (12 = максимум 1 час за итерацию), чтобы не блокировать воркер надолго.

// Пример: checkpoint = 14:30, now = 14:48, safetyLag = 30s // cutoff = floor5min(14:47:30) = 14:45 // pending windows: [14:30–14:35] [14:35–14:40] [14:40–14:45]

Process Each Window

Для каждого окна вызываем WindowBucketBuildService.BuildWindowAsync(window)
(подробности — в секции 3). После каждого окна обновляем чекпоинт и продлеваем lease.

Release Lease & Repeat

Возвращаемся к Sleep. Если окна закончились — воркер idle до следующего poll.

3 Обработка одного окна (BuildWindowAsync)

1. Read Source Events

IPathSourceEventReader.GetWindowAsync(window)
SELECT из ClickHouse: все события с occurred_at в окне [start, end).
Каждое событие с N process_keys → N строк PathSourceEvent.

2. Load Existing Cursors

IChainCursorRepository.GetByChainKeysAsync(chainKeys)
Для каждого уникального ChainKey из событий подгружаем курсор — последнее событие этой цепочки из предыдущего бакета.

3. Aggregate (in-memory)

IPathAggregationService.Aggregate(window, events, cursors)
Дедупликация → группировка по chain → chain stitching → сборка NodeBuckets + EdgeBuckets

4. Persist (upsert to MongoDB)

PathNodeBucketRepository.UpsertBatchAsync()
PathEdgeBucketRepository.UpsertBatchAsync()
ChainCursorRepository.UpsertBatchAsync()
Upsert = если бакет с таким ключом есть — заменяем. Повторная обработка безопасна.

4 Агрегация: от событий к бакетам
// Одно raw event с 2 process_keys порождает 2 PathSourceEvent. // Но event_id одинаковый → дедуплицируем по (event_id + chain_key). events.DistinctBy(e => new SourceEventDedupKey(e.EventId, e.ChainKey))
Зачем? Если event-processor записал одно событие дважды (at-least-once delivery), или одно событие несёт несколько process_keys с одинаковым ChainKey — мы не хотим считать его дважды в одном бакете.
// ChainKey = (org_id, project_id, process_def_id, process_id) // Каждая группа — это одна пользовательская цепочка (сессия/заказ/etc). events.GroupBy(e => e.ChainKey) .Select(g => g.OrderBy(e => e.OccurredAtUtc) .ThenBy(e => e.EventId))
Внутри группы события сортируются по времени. Если два события произошли в одну миллисекунду — fallback на EventId для детерминированного порядка.

Для каждой цепочки ChainStitchingService строит NodeVisits и EdgeTransitions:

// Для каждого события в цепочке: for (i = 0; i < events.Count; i++) { // 1. NodeVisit nodeVisits.Add(new NodeVisit( chainKey, nodeKey, IsEntry: i == 0 && cursor == null, // первое событие без курсора = вход IsExit: i == last // последнее в окне = выход (приблизительно) )); // 2. Edge: cursor → first event (сшивка через границу!) if (i == 0 && cursor != null) edges.Add(cursor.LastEvent → events[0]); // 3. Edge: event[i-1] → event[i] (внутри окна) if (i > 0) edges.Add(events[i-1] → events[i]); } // 4. Обновить курсор последним событием окна cursor = new ChainCursor(events[last]);
// Группируем NodeVisits по NodeKey, считаем метрики: nodeVisits.GroupBy(v => v.Key) .Select(g => new PathNodeBucket( Key: g.Key, VisitsCount: g.Count(), EntriesCount: g.Count(v => v.IsEntry), ExitsCount: g.Count(v => v.IsExit), UniqueChains: g.Select(v => v.ChainKey).Distinct().Count() )); // Аналогично для EdgeTransitions → PathEdgeBucket
5 Chain Stitching — сшивка путей через границы бакетов

Пользователь выполняет 4 действия. Два попадают в бакет 14:30, два — в бакет 14:35. Без stitching мы теряем переход product_click → add_to_cart.

Бакет 14:30 – 14:35 cursor = null
page_view → product_click
Nodes: page_view (entry), product_click (exit*)
Edges: page_view → product_click
Cursor → product_click @ 14:34
cursor
Бакет 14:35 – 14:40 cursor = product_click
product_click → add_to_cart → checkout
Nodes: add_to_cart (not entry!), checkout (exit*)
Edges: product_click → add_to_cart, add_to_cart → checkout
Cursor → checkout @ 14:38
Фиолетовый — данные из курсора (предыдущий бакет). Переход product_click → add_to_cart восстановлен благодаря stitching, хотя события разнесены по разным бакетам.

8 End-to-End: от клика пользователя до графа

SDK (браузер)

Пользователь кликает. SDK отправляет событие с eventName, processKeys, actor, occurredAt.

Ingest Service

Валидирует, нормализует, публикует EventMessageDto в RabbitMQ.

Event Processor

Consume из RabbitMQ → bulk INSERT в ClickHouse (таблица events).

Bucket Builder (этот сервис)

Polling: читает окно из ClickHouse → агрегирует nodes/edges → upsert в MongoDB.
Stitching через курсоры сохраняет непрерывность путей.

Analytics API

Читает бакеты из MongoDB, мёрджит за запрашиваемый период → JSON-ответ.

Frontend (React Flow)

Рисует интерактивный граф: узлы = события, рёбра = переходы, метрики на каждом элементе.
